
# 开发规范指南
为保证代码质量、可维护性、安全性与可扩展性，请在开发过程中严格遵循以下规范，本规范完全适配当前项目实际环境。

## 一、项目基础信息
1. **用户工作目录**：`D:\smart-streetlight`，所有项目文件、开发操作均需在该目录下进行，禁止随意修改工作目录路径
2. **当前目录结构**：
```
smart-streetlight/
└── docs
```
> 注：当前项目仅存在`docs`文档目录，后续开发需按照Maven标准项目结构补充源代码、配置、测试等相关目录，标准参考结构如下：
> ```
> smart-streetlight/
> ├── docs/                # 项目文档目录（已存在）
> ├── src/                 # 源代码目录
> │   ├── main/            # 主代码目录
> │   │   ├── java/        # Java业务源代码
> │   │   └── resources/   # 配置文件、静态资源、模板文件等
> │   └── test/            # 测试代码目录
> │       ├── java/        # 测试用例源代码
> │       └── resources/   # 测试专用配置文件
> ├── pom.xml              # Maven项目核心配置文件
> └── README.md            # 项目说明文件
> ```
3. **SDK版本**：JDK 1.8.0_131（全局统一，禁止混用其他JDK版本）
4. **构建工具**：Maven（统一使用Maven进行依赖管理、项目构建、打包部署）
5. **代码作者**：侯俊旭

## 二、技术栈要求
- **JDK版本**：1.8.0_131
- **构建工具**：Maven
- **核心依赖**：待项目初始化后根据业务需求统一配置，需遵循最小依赖原则，仅引入必要依赖
- **业务框架**：若后续采用Spring生态框架，需兼容JDK 1.8版本，优先选择Spring Boot 2.x及以下版本

## 三、分层架构规范
| 层级        | 职责说明                         | 开发约束与注意事项                                               |
|-------------|----------------------------------|----------------------------------------------------------------|
| **Controller** | 处理 HTTP 请求与响应，定义 API 接口 | 不得直接访问数据库，必须通过 Service 层调用                  |
| **Service**    | 实现业务逻辑、事务管理与数据校验   | 必须通过 Repository 层访问数据库；返回 DTO 而非 Entity（除非必要） |
| **Repository** | 数据库访问与持久化操作             | 若采用JPA框架需继承 `JpaRepository`；使用 `@EntityGraph` 避免 N+1 查询问题     |
| **Entity**     | 映射数据库表结构                   | 不得直接返回给前端（需转换为 DTO）；包名统一为 `entity`         |

### 接口与实现分离
- 所有业务接口需定义独立接口文件，具体实现类放在接口所在包下的 `impl` 子包中。

## 四、安全与性能规范
### 输入校验
- 使用 `@Valid` 与 JSR-303 校验注解（如 `@NotBlank`, `@Size` 等）
  - 注意：当前JDK版本为1.8，若采用Spring Boot 2.x及以下版本，校验注解位于 `javax.validation.constraints.*`；若后续升级框架版本，需同步调整校验注解包路径
- 禁止手动拼接 SQL 字符串，防止 SQL 注入攻击；若采用ORM框架，需使用参数化查询。

### 事务管理
- `@Transactional` 注解仅用于 **Service 层**方法。
- 避免在循环中频繁提交事务，影响性能。

## 五、代码风格规范
### 命名规范
| 类型       | 命名方式             | 示例                  |
|------------|----------------------|-----------------------|
| 类名       | UpperCamelCase       | `UserServiceImpl`     |
| 方法/变量  | lowerCamelCase       | `saveUser()`          |
| 常量       | UPPER_SNAKE_CASE     | `MAX_LOGIN_ATTEMPTS`  |

### 注释规范
- 所有类、方法、字段需添加 **Javadoc** 注释，统一使用中文编写（与项目第一语言保持一致），专有名词、技术术语可保留英文。

### 类型命名规范（阿里巴巴风格）
| 后缀 | 用途说明                     | 示例         |
|------|------------------------------|--------------|
| DTO  | 数据传输对象                 | `UserDTO`    |
| DO   | 数据库实体对象               | `UserDO`     |
| BO   | 业务逻辑封装对象             | `UserBO`     |
| VO   | 视图展示对象                 | `UserVO`     |
| Query| 查询参数封装对象             | `UserQuery`  |

### 实体类简化工具
- 使用 Lombok 注解替代手动编写 getter/setter/构造方法：
  - `@Data`
  - `@NoArgsConstructor`
  - `@AllArgsConstructor`

## 六、扩展性与日志规范
### 接口优先原则
- 所有业务逻辑通过接口定义（如 `UserService`），具体实现放在 `impl` 包中（如 `UserServiceImpl`），降低模块耦合度。

### 日志记录
- 使用 `@Slf4j` 注解代替 `System.out.println`，日志级别遵循ERROR > WARN > INFO > DEBUG的优先级，禁止打印敏感信息。

## 七、依赖管理规则
1. 所有依赖统一在项目根目录的`pom.xml`中声明，禁止在代码中硬编码依赖路径
2. 多模块项目需在父模块的`<dependencyManagement>`标签中统一管理依赖版本，避免子模块版本冲突
3. 遵循最小依赖原则，仅引入业务必须的依赖，定期清理无用依赖，减小项目体积
4. 第三方依赖需从Maven中央仓库或公司内部私有仓库拉取，禁止引入未经验证的开源依赖
5. 依赖版本升级需提前评估兼容性，提交变更说明，禁止随意升级核心依赖版本

## 八、编码原则总结
| 原则       | 说明                                       |
|------------|--------------------------------------------|
| **SOLID**  | 高内聚、低耦合，增强可维护性与可扩展性     |
| **DRY**    | 避免重复代码，提高复用性                   |
| **KISS**   | 保持代码简洁易懂                           |
| **YAGNI**  | 不实现当前不需要的功能                     |
| **OWASP**  | 防范常见安全漏洞，如 SQL 注入、XSS 等      |
